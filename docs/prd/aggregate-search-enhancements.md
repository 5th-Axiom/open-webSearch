# 聚合搜索增强 PRD

## 项目背景

`open-websearch` 已经支持多个搜索引擎并发查询，并通过 MCP、CLI 和本地 HTTP daemon 对外提供搜索能力。当前聚合逻辑较轻量：按全局 `limit` 在多个引擎之间分配配额，将结果拼接后返回。这个行为可用，但在多引擎同时返回相同页面时会产生重复结果，也缺少跨引擎排序、权重和深度检索控制。

## 目标用户

- 使用 MCP 客户端进行实时网页检索的 Agent 用户
- 通过本地 daemon HTTP API 集成搜索能力的开发者
- 使用 CLI 做一次性搜索或脚本自动化的本地用户

## 核心目标

- 支持 URL 归一化去重，减少多引擎重复结果。
- 支持相同 URL 的多引擎来源合并。
- 支持 Reciprocal Rank Fusion 排名融合。
- 支持按搜索引擎配置权重。
- 支持每个搜索引擎独立候选数量控制。
- 支持 `fast`、`balanced`、`deep` 三种聚合模式。
- 保持 MCP、CLI、HTTP daemon 入口使用同一套核心搜索服务。

## 功能范围

- 核心搜索服务增加聚合选项和结果融合。
- HTTP daemon `/search` 接受聚合参数。
- CLI `search` 命令接受聚合参数。
- MCP `search` 工具接受聚合参数。
- 单元测试覆盖去重、融合排序、权重、每引擎配额和参数解析。
- 文档补充 HTTP API 与 CLI 用法。

## 非目标范围

- 不新增搜索引擎。
- 不引入外部排序服务或机器学习重排。
- 不改变已有 fetch 类接口。
- 不实现持久化缓存或跨请求结果缓存。
- 不改造 Docker 发布流程。

## 用户流程

1. 用户选择多个搜索引擎。
2. 用户可选配置聚合模式、每引擎候选数、排序方式、去重开关和权重。
3. 系统并发请求各搜索引擎。
4. 系统对返回 URL 做归一化。
5. 系统合并重复 URL，并记录参与命中的搜索引擎。
6. 系统按聚合模式和排序策略输出最终结果。
7. 若部分引擎失败，系统仍返回成功引擎结果，并在 `partialFailures` 中保留失败信息。

## 核心功能列表

- `aggregationMode`: `fast`、`balanced`、`deep`
- `perEngineLimit`: 每个引擎最多请求多少候选结果
- `ranking`: `engine-order` 或 `rrf`
- `engineWeights`: 每个引擎的排序权重
- `dedupe`: 是否启用 URL 归一化去重
- 聚合结果保留原有字段，并可附加 `engines` 和 `score`

## 技术约束

- 默认行为必须尽量兼容现有调用方。
- 不新增运行时依赖。
- 核心逻辑放在 `src/core/search`。
- 参数校验沿用现有 CLI、HTTP、MCP 风格。
- `limit` 仍表示最终返回结果数，`perEngineLimit` 只控制候选池大小。
- 单个引擎请求上限保持在 `1-50`。

## 验收标准

- 多引擎重复 URL 会被归一化并合并。
- 合并结果包含命中的搜索引擎列表。
- RRF 排序会提升多个引擎共同命中的结果。
- 引擎权重会影响 RRF 排序。
- `perEngineLimit` 能让每个引擎独立取 N 条候选。
- `fast`、`balanced`、`deep` 模式有清晰且可测试的候选配额策略。
- CLI、HTTP daemon、MCP 三个入口都能传入聚合参数。
- 现有核心搜索测试和 CLI 测试通过。

## 风险与待确认问题

- 某些站点的 URL 参数可能有业务含义，过度归一化可能误合并结果。本版本只移除常见追踪参数并保守规范化。
- 不同搜索引擎返回的标题和摘要质量不同，合并时暂时保留排序胜出候选的标题和摘要。
- `linuxdo` 在 README 中标注暂不支持，聚合增强不改变该状态。
- 默认模式采用兼容优先策略，深度聚合需显式指定。

## 模块拆解

- 核心模块：搜索聚合选项、URL 归一化、重复合并、RRF 排序。
- 数据模型：`SearchResult` 增加可选聚合元数据。
- HTTP API：`POST /search` 请求体增加聚合参数。
- CLI：`open-websearch search` 增加聚合参数。
- MCP：`search` 工具 schema 增加聚合参数。
- 测试模块：核心服务测试、CLI 参数测试、daemon 行为测试。
- 文档模块：HTTP API、README 搜索示例。

## 开发阶段

### Phase 1：核心聚合能力

目标：在核心搜索服务中实现 URL 归一化、去重、RRF、权重和配额策略。

产出：核心类型、聚合函数和单元测试。

验收：核心搜索测试覆盖去重、排序、权重和 `perEngineLimit`。

### Phase 2：入口参数接入

目标：HTTP daemon、CLI、MCP 均可传入聚合选项。

产出：参数解析、校验、schema 更新和相关测试。

验收：三个入口共用核心能力，参数非法时返回现有风格错误。

### Phase 3：文档与回归验证

目标：补充使用说明并运行相关测试。

产出：HTTP API 文档和 README 示例更新。

验收：测试通过，文档包含多引擎聚合示例。
